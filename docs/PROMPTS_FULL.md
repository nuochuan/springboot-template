# PROMPTS.md

本文档用于在完成一个可运行、可测试的小型项目。

项目默认技术栈：

* Java 8
* Spring Boot 2.7.x
* Gradle
* JUnit 5
* 本地运行
* 不依赖数据库、Redis、MQ、配置中心，除非任务明确要求

核心原则：

1. 先定义 MVP，再写代码。
2. 先写需求、测试、计划，再让 AI 实现。
3. AI 负责提效，人负责需求、边界、正确性和取舍。
4. 代码生成后必须 review diff。
5. 测试通过不代表业务正确，仍需人工检查核心逻辑。
6. 修 bug 时必须先复现、再分析根因、再做最小修复。
7. 最终交付必须包含 README、测试结果和 Demo 说明。

# 1. 开始任务时的第一条 Prompt

适用场景：刚拿到一个开放编码题，还没有写任何代码。

```text
请先阅读以下文件：

- AGENTS.md
- REQUIREMENTS.md
- TEST_CASES.md
- PLAN.md

当前任务是一个编码任务。请先不要写代码。

请你先做三件事：

1. 检查 REQUIREMENTS.md 是否清楚定义了目标、输入、输出、核心规则、边界情况和暂不实现内容。
2. 检查 TEST_CASES.md 是否覆盖正常场景、边界场景和异常场景。
3. 检查 PLAN.md 是否能支持一个 1 小时内可完成的 MVP。

要求：
- 只指出缺失点和风险点。
- 不要创建 Java 文件。
- 不要修改代码。
- 不要添加数据库、Redis、MQ、登录、复杂前端。
- 输出请用中文。
```

# 32. 不要使用的 Prompt

以下 Prompt 不推荐使用：

```text
帮我写一个完整项目。
```

问题：范围失控，AI 容易乱加功能。

```text
帮我实现一下这个功能。
```

问题：没有需求、边界和验收标准。

```text
测试失败了，帮我修。
```

问题：缺少输入、期望、实际和日志，AI 只能猜。

```text
帮我优化一下。
```

问题：优化目标不明确，容易大范围重构。

```text
你觉得怎么做最好？
```

问题：把决策权交给 AI，失去工程控制感。

更好的方式是：

```text
请基于 REQUIREMENTS.md、TEST_CASES.md、PLAN.md，只实现 MVP。
```

```text
请先解释根因，再给出最小修复方案。
```

```text
请 review diff，检查是否有超出 MVP 的内容。
```

```text
请检查 Java 8 兼容性，不要改变业务逻辑。
```

# 33. 最终原则

限时编码任务不是比谁让 AI 写得快，而是比谁能稳定交付。

你要始终控制四件事：

1. 需求边界
2. 实现范围
3. 验证方式
4. 交付表达

AI 可以生成代码，但不能替你负责正确性。

最终交付时，至少要能回答：

1. 这个项目实现了什么？
2. 哪些内容明确没有做？
3. 正确性如何验证？
4. 核心边界如何处理？
5. 如果继续扩展，下一步做什么？
