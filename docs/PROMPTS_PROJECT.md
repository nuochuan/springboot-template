# AI Coding 流程提示词

本文档用于完整的工作流，覆盖从需求澄清、方案设计、编码实现、测试验证到交付复盘的全过程。

## 1. 工作原则

- 先理解目标，再修改代码。
- 先定位上下文，再提出方案。
- 小步提交，小步验证。
- 所有实现必须有可验证结果。
- 不确定的地方先暴露假设，不静默猜测关键业务规则。
- 不扩大修改范围，只改与当前任务直接相关的文件。
- 保留用户已有改动，不覆盖未授权内容。

## 2. 标准流程总览

```text
需求输入
  ↓
需求澄清
  ↓
上下文收集
  ↓
方案设计
  ↓
任务拆解
  ↓
编码实现
  ↓
自测验证
  ↓
代码审查
  ↓
交付说明
  ↓
复盘优化
```

## 3. 阶段一：需求澄清

目标：确认要解决的问题、边界、验收标准。

推荐提示词：

```text
你现在是资深软件工程师。请先不要写代码，先帮我澄清需求。

任务背景：
{描述业务背景}

我要实现：
{描述目标功能}

请输出：
1. 你对需求的理解
2. 需要确认的问题
3. 默认假设
4. 功能边界
5. 可验收标准
```

如果需求已经明确，可以使用：

```text
基于以下需求，请直接给出实现边界和验收标准，不要开始编码。

需求：
{需求内容}
```

## 4. 阶段二：上下文收集

目标：让 AI 先读项目结构、相关文件、依赖和现有实现，避免凭空生成代码。

推荐提示词：

```text
请先分析当前项目，不要修改任何文件。

请重点查看：
1. 项目结构
2. 构建工具和运行方式
3. 相关模块、类、接口
4. 已有测试
5. 可能受影响的文件

最后输出：
1. 项目当前状态
2. 与本任务相关的关键文件
3. 推荐修改范围
4. 风险点
```

限制修改范围时使用：

```text
只允许分析和修改以下文件：
{文件列表}

如果发现必须修改其他文件，请先说明原因并等待确认。
